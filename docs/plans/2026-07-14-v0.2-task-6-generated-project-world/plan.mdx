## Outcome

Task 6 turns a registered generated game into a restart-safe Project World without regenerating the project or implying that arbitrary generated quests can be implemented. Done means **Last-Moment Pulse** opens from Launchpad, renders only validated persisted artifacts, launches the verified starter in Godot, opens its folder, atomically saves an idea seed with an activity note shown in the Chronicle view, and restores the same project and selected quest after restart.

<Callout id="truth-decision" tone="decision">

**Separate verified starter truth from blueprint intent.** The current playable state may claim only what persisted Godot verification proved: the bounded arena, keyboard movement, objective relay, required scripts/nodes, and code-native project load. The visual is labelled **Verified starter layout** / **Playable-state preview** and is a static, verification-derived layout. Enemy approach and push pulse remain roadmap outcomes. Quest briefs label their text as the generated **intended outcome**; Task 6 does not infer a trustworthy code delta or hide starter overlap. `chronicle.json` remains authoritative for real Chronicle events, while idea-seed activity appears only as derived activity in the Chronicle view.

</Callout>

## Scope and non-goals

In scope: Launchpad recent-project integration; generated Project World; roadmap and quest brief rendering; Chronicle and documentation disclosure; idea-seed persistence; selected view/quest reload; Godot launch; folder opening; responsive and accessible behavior; browser, service, API, and protected regression evidence.

Deferred: generated-quest implementation or regeneration, additional starters, existing-project import, Graphify, Godot MCP, general chat, project scanning, sample art, default-command changes, and submission packaging.

Existing Task 5 projects treat an absent idea file as an empty list.

</Callout>

## New and reused host actions

All paths are resolved from a registered `projectId`; clients never submit filesystem paths, commands, Godot executables, quest state, or arbitrary document names.

Opening from Launchpad uses **`POST /api/projects/:projectId/open`**. It accepts only the path `projectId`, resolves and validates the registered project, atomically updates registry `lastOpenedAt` only after validation succeeds, and returns the validated Project World snapshot. **`GET /api/projects/:projectId/world` is strictly read-only** and is used for reloads of an already-open world without changing recency. Selected-view persistence remains a separate same-origin `POST /api/projects/:projectId/state` action.

## Persistence, failure, and empty behavior

Opening from Launchpad calls `POST /api/projects/:projectId/open`. The action resolves the registry entry, canonicalizes the registered directory, verifies it is still the same direct child of Forge's projects root, reads the manifest, resolves only manifest-owned relative artifacts, validates every artifact and cross-reference, then atomically updates registry `lastOpenedAt` and returns the validated snapshot. Validation failure performs no recency write.

`GET /api/projects/:projectId/world` repeats the validated read for an already-open Project World and is strictly read-only. Reloading cannot update `lastOpenedAt`, selected state, Chronicle, ideas, roadmap, verification, Git, or registration.

Quest selection writes only `project-state.json` through its explicit state mutation. Returning to Launchpad is runtime navigation; it does not erase the last selected quest. Reopening after restart uses the same recent-project ID and project-local state—no creation service, GPT call, starter copy, Godot verification rewrite, Git operation, or registry duplication runs. An absent `.forge/idea-seeds.json` is the valid empty state; it is created only by the first successful idea save. `chronicle.json` stays authoritative for real Chronicle events; idea activity is derived from the idea-seed record and labelled as activity in the merged feed.

## Responsive and accessibility requirements

Desktop keeps the verified starter layout / playable-state preview anchored beside a visually dominant horizontal roadmap. Tablet stacks the identity and starter world preview above the roadmap while preserving the four-node left-to-right order with controlled horizontal overflow only inside the roadmap rail. At narrow width the roadmap becomes one ordered vertical sequence with dependencies between adjacent nodes; the Companion sits below the recommended node; Chronicle and documentation follow the roadmap; the idea composer becomes a full-width stacked dock; quest brief sections become a single reading column; Godot/folder actions remain visible without covering content.

Use semantic landmarks and heading order, real buttons, `aria-current="step"` on the selected quest, labelled dependency/status text in addition to color, a polite live region for persisted/launch results, error alerts, visible focus, logical tab order, minimum control targets, contrast-compliant ember/cyan/violet/mint tokens, and `prefers-reduced-motion` behavior. Focus returns to the triggering roadmap node after closing a brief and moves to the saved seed status after a successful idea write.

## Recommended implementation sequence

1. **Contracts and join rules:** extend only generated-project state/Chronicle contracts; define the UI-safe snapshot and cross-artifact validation.
2. **Read-only adapter:** implement registered-path resolution and full artifact load behind the strictly read-only `GET /world`; prove reload changes no recency or project bytes.
3. **Read-only UI:** enrich the Launchpad card, add Generated Project World, roadmap, Chronicle/docs disclosure, and quest brief with honest implementation messaging.
4. **Server-controlled actions:** add the explicit `POST /open` Launchpad mutation, pinned-Godot launch, and the canonical open-folder boundary.
5. **Persistence:** atomically write selection state and one idea-seeds artifact; show its clearly labelled derived activity note in the Chronicle view; reload with a fresh service instance.
6. **Responsive, accessibility, and failure states:** add narrow ordering, focus recovery, reduced motion, invalid/missing project handling, and no-optimism errors.
7. **Evidence and closeout:** focused tests, protected v0.1 checks, real Last-Moment Pulse restart/launch rehearsal, five-state Edge review, diff review, roadmap/handoff/work-log updates, and one task-scoped commit.

## Rollback strategy

<Callout id="rollback" tone="risk">

Keep the Task 5 **Project Created** summary as the fallback surface until Task 6 closes. Generated-world routes and components are additive under `npm run forge:v0.2`; `npm run forge`, `npm run forge:v0.1`, sample endpoints, and the controlled creation transaction stay untouched. A code rollback removes the new adapter/routes/component and restores the summary action. Task 6's broader project-state view and optional idea-seeds file remain ignored by the Task 5 summary loader, so rollback does not require deleting user data. Never roll back by deleting a registered project or rewriting its roadmap.

</Callout>
